# Git Workflow Rules — MANDATORY for AI Code Changes

> **LOAD**: When AI writes production code | **PURPOSE**: Enforce proper git workflow
> **PREREQUISITE**: `CORE.md` must be loaded first
> **SKILLS**: `git` (commit/push/pr/merge), `using-git-worktrees`, `finishing-a-development-branch`

---

## GOLDEN RULE

**AI MUST NEVER commit code directly to `main` or `develop`.**
All code changes go through feature branches → Pull Request → Review → Merge.

---

## 1. Branch Strategy

### Branch Naming Convention

```
<type>/<descriptive-kebab-case-name>
```

| Type | Purpose | Example |
|------|---------|---------|
| `feature/` | New features | `feature/customer-debt-offset` |
| `fix/` | Bug fixes | `fix/subscription-expiry-check` |
| `refactor/` | Code restructure | `refactor/warehouse-service-split` |
| `docs/` | Documentation only | `docs/api-reference-update` |
| `test/` | Test improvements | `test/integration-payment-flow` |
| `chore/` | Maintenance, config | `chore/update-deps` |
| `hotfix/` | Production critical fix | `hotfix/auth-token-leak` |

### Protected Branches (NEVER direct push)

- `main` — production
- `develop` — staging (if used)
- `release/*` — release candidates

### Branch Flow

```
main (production-ready)
  └─ feature/ai-dev (AI development base)
       └─ feature/xxx (feature branches — AI creates these)
            └─ PR → feature/ai-dev → review → merge
```

## 2. AI Code Change Workflow (MANDATORY)

### When AI is asked to implement/fix/refactor code:

```
Step 1: CREATE BRANCH
  ├─ Checkout from base branch (feature/ai-dev or as specified)
  ├─ Name: <type>/<descriptive-name>
  └─ Prefer git worktree for isolation (activate using-git-worktrees skill)

Step 2: IMPLEMENT
  ├─ Write code following development-rules.md
  ├─ Commit frequently with conventional commits
  ├─ Each commit = one logical change
  └─ Run linting + tests before each commit

Step 3: VERIFY
  ├─ All tests pass (run fresh, read output)
  ├─ No secrets in diff (security scan)
  ├─ Code compiles/builds without errors
  └─ No regressions

Step 4: CREATE PULL REQUEST
  ├─ Push branch to remote
  ├─ Create PR via `gh pr create` using .github/PULL_REQUEST_TEMPLATE.md
  ├─ Fill ALL required sections (no blanks)
  ├─ Assign reviewer (or notify team)
  └─ Report PR URL to user

Step 5: WAIT FOR REVIEW
  ├─ Do NOT merge own PR
  ├─ Address review feedback if any
  └─ Re-push fixes, re-request review

Step 6: COMPLETE (after approval)
  ├─ Merge via PR (squash or merge commit per team preference)
  ├─ Delete feature branch after merge
  └─ Cleanup worktree if used (activate finishing-a-development-branch skill)
```

### Exception: Documentation-only changes

Documentation, plans, and config changes (no production code) MAY be committed directly to the current branch if user explicitly requests. Still follow conventional commits.

### Exception: User explicit override

If user says "commit directly" or "push to main", comply but WARN:
```
⚠️ Direct commit to {branch} bypasses PR review.
Proceed? (y/n)
```

## 3. Commit Standards

### Format

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Types

| Type | When |
|------|------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `refactor` | Code change that neither fixes nor adds feature |
| `test` | Adding or correcting tests |
| `chore` | Maintenance, deps, config |
| `perf` | Performance improvement |
| `style` | Formatting, no code change |

### Scope (use service or module name)

```
feat(core): add workspace invitation API
fix(gara-task): correct LSC status transition
docs(datatool): update ingest pipeline architecture
refactor(warehouse): split inventory service
```

### Rules

- No AI references in commit messages (no "Generated by Claude", no "Co-Authored-By: AI")
- Keep subject line < 72 characters
- Use imperative mood: "add feature" not "added feature"
- One logical change per commit
- Never commit: `.env`, API keys, credentials, `node_modules/`, build artifacts

## 4. Pull Request Standards

### Before Creating PR

```
PRE-PR CHECKLIST:
  □ Tests pass (fresh run, not cached)
  □ No secrets in diff
  □ Code compiles/builds
  □ Commits are clean (squash fixup commits)
  □ Branch is up-to-date with base (rebase if needed)
  □ PR title uses conventional format
```

### PR Content (from .github/PULL_REQUEST_TEMPLATE.md)

1. **Problem statement** — what and why
2. **What changed** — 1-3 sentences
3. **Affected services** — checklist
4. **Breaking changes** — if any
5. **Test evidence** — mandatory, not just "it works"
6. **Checklist** — all items checked

### PR Size Guidelines

| Size | Files | Action |
|------|-------|--------|
| Small | ≤ 5 | Single PR |
| Medium | 6-15 | Single PR, ensure focused |
| Large | > 15 | SPLIT into multiple PRs |

**Split strategy**: by service boundary, by logical concern, or by layer (db → api → ui).

## 5. Security Protocols

### Pre-Commit Scan (MANDATORY)

```bash
# Scan staged files for secrets
git diff --cached | grep -iE "(AKIA|api[_-]?key|token|password|secret|credential|private[_-]?key|mongodb://|postgres://|mysql://|redis://|-----BEGIN)"
```

**If secrets detected**: BLOCK commit, warn user, suggest `.gitignore` or env vars.

### Files NEVER Committed

- `.env`, `.env.*` (except `.env.example`)
- `*.key`, `*.pem`, `*.p12`
- `credentials.json`, `secrets.json`
- `node_modules/`, `dist/`, `build/`
- `.DS_Store`

### Force Push Rules

- **NEVER** force push to protected branches (`main`, `develop`, `release/*`)
- **ALLOWED** on feature branches after rebase (with user confirmation)

## 6. Red Flags

| Thought | Reality |
|---------|---------|
| "I'll just commit to main, it's a small change" | ALL changes go through PR |
| "No need for a branch, just a quick fix" | Even hotfixes need branches |
| "I'll create the PR later" | PR is part of the workflow, not optional |
| "Tests take too long, skip them" | No commit without passing tests |
| "This secret is just for local dev" | NEVER commit secrets |
| "I'll skip the PR template, it's obvious" | Fill ALL sections |
| "I can merge my own PR" | Wait for review unless explicitly told otherwise |

## Quick Reference

```
USER: "implement feature X"

AI WORKFLOW:
  1. git checkout -b feature/x (from base)
  2. [implement + test + commit]
  3. git push -u origin feature/x
  4. gh pr create --base feature/ai-dev --title "feat(scope): ..."
  5. Report PR URL to user
  6. Wait for review approval
  7. Merge + cleanup
```
